# قانون «اول کامپوننت‌های موجود، بعد کد سفارشی»

این قانون می‌گوید: **قبل از نوشتن هر UIِ سفارشی، اول باید چک کنی که آیا پروژه از
قبل کامپوننتی برای این کار دارد یا نه.** اگر دارد، از همان استفاده کن. نوشتن یک
اینپوت/دکمه/مودال/جدولِ جدید با استایل متفاوت، درحالی‌که نسخه‌ی استاندارد پروژه
موجود است، ممنوع.

## چرا این قانون مهم است

وقتی هر کس برای هر نیاز کوچک یک المان خام (`<input>`, `<button>`, یک `<div>` با
کلی کلاس tailwind) از صفر می‌سازد، نتیجه‌اش این می‌شود:

- **ناهماهنگی بصری:** ده‌جور اینپوت با ده‌جور padding/border/رنگ/حالت focus در
  اپ ظاهر می‌شود و ظاهرِ محصول تکه‌تکه به‌نظر می‌رسد.
- **تکرار منطق:** هندل کردن خطا، حالت loading، حالت disabled، RTL/LTR، و
  دسترس‌پذیری، هر بار از نو و ناقص پیاده می‌شود.
- **هزینه‌ی نگه‌داری:** یک تغییر طراحیِ ساده (مثلاً گرد شدن گوشه‌ها) باید در
  ده‌ها المانِ دست‌ساز جداگانه اعمال شود.
- **باگ‌های رفتاری:** کامپوننت مشترک پروژه edge caseها را از قبل حل کرده؛ نسخه‌ی
  دست‌ساز همه را دوباره باگ‌دار می‌کند.

راه‌حل: **همیشه اول از design system داخلی پروژه استفاده کن.**

---

## قبل از نوشتن، اول جست‌وجو کن

پروژه یک مجموعه کامپوننت مشترک در `src/app/_components/common/` دارد (با پیشوند
`Taban`) و کامپوننت‌های عمومی دیگری در `src/app/_components/`. **قبل** از ساختن
هر المانِ تعاملی یا نمایشی:

1. در `src/app/_components/common/` بگرد که آیا کامپوننت متناظر وجود دارد.
2. اگر نبود، در کل `src/app/_components/` جست‌وجو کن.
3. اگر بازهم نبود و مطمئنی که وجود ندارد، آن‌وقت کامپوننت جدید بساز — ولی آن را
   هم مطابق همین سیستم و به‌صورت قابل‌استفاده‌ی مجدد بساز (نه استفاده‌ی یک‌بارِ
   inline).

کامپوننت‌های مشترکِ موجود (نمونه — قبل از ساختِ هرکدام از این‌ها حتماً از نسخه‌ی
آماده استفاده کن):

- ورودی متن → `TabanInput` (نه `<input>` خام)
- دکمه → `TabanButton` (نه `<button>` خام؛ variantها و `isLoading`/`isLink` دارد)
- چک‌باکس → `TabanCheckbox`
- تکست‌اریا → `TabanTextarea`
- رادیو → `TabanRadioGroup`
- انتخاب‌گر/اتوکامپلیت → `TabanAutoComplete`, `TabanAutocompleteWrapper`,
  `TabanAutoCompleteCheckbox`
- مودال → `TabanModal`
- آکاردئون → `TabanAccordion`
- لودینگ → `TabanLoading`
- جدول → `TabanTable`
- آپلود فایل → `fileUploader` / `uploadBox`

> این فهرست نمونه است و ممکن است تغییر کند — همیشه پوشه‌ها را واقعاً نگاه کن،
> به این لیست به‌عنوان مرجع نهایی تکیه نکن.

```tsx
// ❌ بد — اینپوت خام با استایل دلخواه، درحالی‌که TabanInput موجود است
<input
	className="border border-gray-300 rounded-md px-3 py-2 focus:ring-2 ..."
	value={name}
	onChange={(e) => setName(e.target.value)}
/>

// ✅ خوب — استفاده از کامپوننت استاندارد پروژه
<TabanInput
	label="نام"
	name="firstName"
	groupMode
	setValue={setFormValues}
	value={formValues.firstName}
	isHandleError
	hasError={!!findError(formErrors, "firstName")}
	errorText={findError(formErrors, "firstName")?.message}
/>
```

## اگر کامپوننت موجود کم داشت، آن را گسترش بده — نکن یکی جدید بساز

اگر کامپوننت استاندارد پروژه دقیقاً نیاز تو را ندارد (مثلاً یک variant یا یک prop
جدید لازم داری):

- **اول**، همان کامپوننت مشترک را با یک prop/variant جدید گسترش بده تا همه از آن
  بهره ببرند.
- **هرگز** یک کپیِ کمی‌متفاوت («`TabanInput2`» یا یک اینپوت دست‌ساز با استایل
  اندکی فرق‌دار) نساز.
- استایل‌دهی از طریق همان API کامپوننت (props / `className` که خودش می‌پذیرد)
  انجام شود، نه با override کردن استایل داخلی‌اش از بیرون.

## چک‌لیست سخت‌گیرانه

قبل از نوشتن هر UI سفارشی این‌ها را چک کن:

1. **گشتم و مطمئن شدم پروژه کامپوننت آماده‌ای برای این کار ندارد؟**
   (`_components/common/` و `_components/`)
2. **دارم از `<input>`/`<button>`/... خام استفاده می‌کنم درحالی‌که معادل
   `Taban…` موجود است؟** → باید عوض شود.
3. **دارم یک نسخه‌ی کمی‌متفاوت از یک کامپوننت موجود می‌سازم؟** → به‌جایش
   کامپوننت موجود را گسترش بده.
4. **اگر واقعاً کامپوننت جدید لازم بود، آن را قابل‌استفاده‌ی مجدد و هم‌سبک با
   بقیه ساختم (نه inline یک‌بار مصرف)؟**

---

## خلاصه‌ی یک‌خطی برای یادآوری سریع

> **قبل از هر کد سفارشی، اول دنبال کامپوننت آماده‌ی پروژه بگرد؛ اگر بود از همان
> استفاده کن، اگر کم داشت گسترشش بده، و فقط وقتی واقعاً نبود یک کامپوننتِ
> قابل‌استفاده‌ی مجددِ هم‌سبک بساز.** ساختِ المان خام یا کپیِ کمی‌متفاوت باید در
> کد ریویو رد شود.
